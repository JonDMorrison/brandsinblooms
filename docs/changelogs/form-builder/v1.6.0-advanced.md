# v1.6.0 Advanced Features And Edge Cases

## Summary

The repository contains a meaningful advanced feature set beyond basic lead capture. Multi-step flows, conditional visibility, uploads, audience enrichment, realtime updates, and operator diagnostics all exist in the current codebase.

What it does not contain is a clean milestone trail tying those features to `FB-011` or `FB-012`. This document records the advanced capability set without inventing milestone ownership.

## Multi-Step Forms

Primary files:

- `src/lib/forms/formFlow.ts`
- `src/components/forms/FormBuildTab.tsx`
- `src/components/forms/DraggableFieldList.tsx`
- `src/components/forms/preview/FormPreviewRenderer.tsx`
- `src/__tests__/formFlow.test.ts`

Confirmed capabilities:

- `FormStep` model with title and description
- per-field `step_index`
- builder toggle for multi-step mode
- step creation and normalization
- publish validation that blocks empty steps
- runtime step navigation in the preview/public renderer
- step progress display and next/previous navigation in the renderer

## Conditional Visibility

Primary files:

- `src/components/forms/DraggableFieldList.tsx`
- `src/lib/forms/formFlow.ts`
- `supabase/functions/_shared/formVisibility.ts`
- `src/__tests__/formVisibility.test.ts`
- `src/__tests__/formFlow.test.ts`

Confirmed capabilities:

- builder UI for rule creation
- supported operators: `equals`, `not_equals`, `contains`, `not_empty`, `is_empty`
- only earlier non-hidden fields can be used as rule sources
- client runtime evaluates rules for rendering
- server sanitization removes inactive field data before storage

This is a fully represented feature across authoring, runtime, and submission sanitization.

## File Uploads

Primary files:

- `src/lib/forms/fileUploads.ts`
- `src/components/forms/DraggableFieldList.tsx`
- `src/components/forms/preview/FormPreviewRenderer.tsx`
- `supabase/functions/submit-form/index.ts`
- `supabase/functions/form-data-cleanup/index.ts`
- `supabase/migrations/20260405113000_form_uploads_bucket.sql`

Confirmed capabilities:

- file field type in the field registry
- builder UI for max files, max file size, and accepted MIME types
- private `form-uploads` storage bucket
- temporary upload paths for public sessions
- permanent upload paths for accepted submissions
- temp-to-permanent finalize flow in `submit-form`
- cleanup worker for stale temp uploads
- runtime guard preventing next/submit while uploads are still active

## Hidden Fields And Mapping Keys

Primary files:

- `src/components/forms/DraggableFieldList.tsx`
- `src/pages/PublicFormPage.tsx`
- `src/lib/forms/submissionPresentation.ts`
- `docs/form-builder-crm-mapping-key.md`

Confirmed behavior:

- hidden field defaults can be authored in the builder
- hidden defaults are injected into outgoing submission payloads
- `mapping_key` is the canonical downstream key for public submissions
- submission presentation resolves by `mapping_key`, field `id`, and legacy fallback keys

## Audience Enrichment

Primary files:

- `src/components/forms/FormAudienceTab.tsx`
- `src/hooks/useFormEditor.ts`
- `src/hooks/useForms.ts`
- `supabase/functions/submit-form/index.ts`

Confirmed behavior:

- forms can request persona assignment
- forms can request tag assignment
- the submit pipeline applies personas and tags after an accepted submission
- audience-processing debug metadata is stored in submission metadata
- the audience tab also states that segment evaluation is always active

Important honesty note:

- this is post-submit enrichment, not public access control
- nothing in the current public route prevents a visitor from loading a published form based on audience settings alone

## Realtime And Diagnostics

Primary files:

- `src/hooks/useFormSubmissionsRealtime.ts`
- `src/components/forms/FormSubmissionsTab.tsx`
- `src/components/forms/FormTestMatrix.tsx`

Confirmed capabilities:

- realtime `INSERT` subscription for `form_submissions`
- queued update handling when the tab is not visible
- in-product developer test matrix for consent, spam, rate-limit, and happy-path checks

## What Still Looks Missing Or Deferred

- No form versioning or history browser
- No A/B testing surface
- No direct webhook builder UI
- No builder UI for embed trigger attributes, despite runtime support
- No enforced `double_opt_in` pipeline
- No repo evidence that cleanly maps these advanced features to `FB-011` or `FB-012`
