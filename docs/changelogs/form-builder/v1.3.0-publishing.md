# v1.3.0 Publishing, Sharing, And Public Runtime

## Summary

Publishing is split across two layers:

1. The internal share and validation UI in the React app.
2. The public embed runtime shipped as static assets under `public/forms/`.

This document focuses on how forms become publicly reachable and how the runtime evolved over time.

## Publish Validation And Share UI

Primary files:

- `src/components/forms/FormPublishTab.tsx`
- `src/lib/forms/publish.ts`
- `src/lib/forms/share.ts`
- `src/pages/crm/FormEditorPage.tsx`

Confirmed publish validation checks:

- form name must exist
- at least one field must exist
- an email field must exist
- SMS consent cannot publish without a phone field
- multi-step forms cannot publish with an empty step

Confirmed share outputs:

- direct public link
- iframe embed code
- JavaScript embed code
- React embed snippet

Confirmed UI-configurable embed options:

- display mode: `inline`, `modal`, `slide-in`
- container selector for inline mode
- trigger button text for modal and slide-in mode

## Public Route And Endpoints

Confirmed public route:

- `/f/:embedKey`

Confirmed public endpoints:

- `GET /functions/v1/get-form-config?embed_key=...`
- `POST /functions/v1/submit-form`

Legacy runtime path still present in the codebase:

- `supabase/functions/serve-embed-js/index.ts`

The React share UI already treats that edge-function runtime as deprecated and prefers the storage-hosted static runtime instead.

### Runtime Evolution

| Runtime | Confirmed evidence | Notable capabilities |
| --- | --- | --- |
| `v1.0.0` | `public/forms/embed.v1.0.0.js` | basic inline runtime, `BloomSuiteForms.init()`, `initForm()` |
| `v1.0.1` | `public/forms/embed.v1.0.1.js` | incremental maintenance version of the basic inline runtime |
| `v1.3.0` | `public/forms/embed.v1.3.0.js` | modal and slide-in display modes, delay/scroll/click triggers, focus trap, accessibility wrappers |
| `v1.4.0` | `public/forms/embed.v1.4.0.js` | MutationObserver support for late-loaded containers, fail-loud UI, diagnostic debug mode, stronger scroll locking |
| `v1.5.0` | `public/forms/embed.v1.5.0.js` | Supabase Storage hosting support and CSS path detection improvements |

The current React share helpers point at the storage-hosted runtime and report `STATIC_EMBED_RUNTIME_VERSION = "1.5.0"` in `src/lib/forms/share.ts`.

## Documentation Drift

There is visible drift between older user-facing docs and the current code path.

Current code reality:

- `src/lib/forms/share.ts` generates URLs from the current app origin and the Supabase storage runtime path

Older docs still reference legacy domains:

- `docs/form-builder-user-guide.md`
- `docs/form-embed-guide.md`

Those docs refer to `forms.bloomsuite.com` and `api.bloomsuite.com`, which do not match the currently generated URLs in the React app.

## Known Gaps

- The builder UI does not expose runtime trigger configuration even though the static runtime supports it.
- The preview tooling shares design intent with the public runtime, but it is not a byte-for-byte reuse of `public/forms/embed*.js`.
- Older documentation still needs to be reconciled with the current storage-hosted runtime strategy.
