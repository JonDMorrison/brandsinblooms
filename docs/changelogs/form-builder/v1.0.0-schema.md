# v1.0.0 Schema And Contract Foundation

## Summary

The Form Builder foundation is larger than the visible `FB-001` milestone. The repository shows an earlier Phase 0 schema rollout, a separate atomic rate-limit RPC, later automation-bridge extensions, and then `FB-001` hardening on top of that base.

This document treats `v1.0.0` as the contract layer: tables, JSON payload shapes, privacy rules, and public endpoint expectations.

## Shipped Foundation

### 1. Core Tables

Primary migration:

- `supabase/migrations/20260128180610_a29e7ba7-f8ee-43ed-9996-771417fff8fd.sql`

This migration creates the core Form Builder tables:

- `public.forms`
- `public.form_submissions`
- `public.form_rate_limits`

It also adds consent detail columns to `public.crm_customers`:

- `email_consent_details`
- `sms_consent_details`

Important schema characteristics:

- `forms.status` is constrained to `draft`, `published`, or `archived`
- `forms.fields_json`, `forms.settings_json`, and `forms.compliance_json` are the main configuration contracts
- `forms.embed_key` is the public identifier used by embeds and direct links
- `form_submissions.result` stores canonical outcomes: `accepted`, `rejected_invalid`, `rejected_rate_limited`, and `rejected_spam`
- `form_rate_limits` is keyed by form, hashed IP, and time window

### 2. Atomic Rate Limiting

Supporting migration:

- `supabase/migrations/20260128184042_6d9f682b-b6d9-4440-9dee-e6720a4357bf.sql`

This adds the `upsert_rate_limit(...)` RPC used by `supabase/functions/submit-form/index.ts` to atomically increment the rate-limit counter.

This matters because the public submission endpoint does not rely on a client-side throttle alone. It uses the database as the real abuse-control source of truth.

### 3. Hardening Layer

Explicit milestone migration:

- `supabase/migrations/20260403120000_fb001_form_builder_contract_hardening.sql`

`FB-001` does not create the schema from scratch. It hardens it by adding and standardizing:

- `form_rate_limits.updated_at`
- `increment_trigger_event_retry(UUID, TEXT)`
- realtime publication for `form_submissions`

This is why `FB-001` should be read as a hardening pass, not the original release of Form Builder storage.

## Type Contracts In The Frontend

Primary contract file:

- `src/types/formBuilder.ts`

Key interface coverage:

- `FormFieldType` includes `email`, `text`, `phone`, `select`, `checkbox`, `file`, `hidden`, `email_consent`, and `sms_consent`
- `FormFieldRules` supports text validation and upload constraints
- `FormVisibilityRule` supports conditional display logic
- `FormStep` supports multi-step flows
- `FormSettings` covers headline copy, layout, theme, success behavior, and `notification_emails`
- `FormCompliance` covers required consent rules plus `double_opt_in` and `gdpr_compliant`
- `FormAudience` covers persona and tag assignment
- `FormSubmissionMetadata` defines the canonical consent and attribution payload shape

Normalization helper:

- `src/lib/forms/designSettings.ts`

This keeps the settings contract stable even when raw JSON is incomplete or malformed.

## Public Endpoint Contracts

### Public Config Fetch

Endpoint implementation:

- `supabase/functions/get-form-config/index.ts`

Contract:

- `GET /functions/v1/get-form-config?embed_key=...`

Behavior confirmed in code:

- validates `embed_key` shape
- loads the form by `embed_key`
- requires `status = 'published'`
- sanitizes output through a strict allowlist before returning it to the browser

Important allowlist behavior:

- layout and theme values are exposed
- `steps` are exposed
- `notification_emails` are not exposed

### Public Submission

Endpoint implementation:

- `supabase/functions/submit-form/index.ts`

Contract:

- `POST /functions/v1/submit-form`

Behavior confirmed in code:

- validates payload shape and `embed_key`
- performs database-backed rate limiting
- performs honeypot spam detection
- validates required fields and consent rules
- records canonical metadata in `form_submissions`
- upserts `crm_customers` without downgrading consent state

## Security And Integrity Guarantees

Confirmed in code and schema:

- RLS exists on `forms`, `form_submissions`, and `form_rate_limits`
- `tenant_id` is derived from the form lookup, not from the public request body
- public config responses use an allowlist instead of a denylist
- client IPs are hashed before storage
- `submit-form` never sets `email_opt_in` or `sms_opt_in` to `false`
- `submit-form` never modifies `opt_out`
- consent snapshots are stored even when consent is not granted

## What Is Explicitly Mapped To Milestones

- `FB-001`: confirmed and shipped

## What Exists But Cannot Be Honestly Mapped To `FB-003` Through `FB-007`

The repository clearly contains shipped schema and contract work outside the visible milestone numbering, but there is no reliable evidence tying that work to `FB-003`, `FB-004`, `FB-005`, `FB-006`, or `FB-007`.

Those IDs are therefore left unmapped on purpose.

## Known Drift And Gaps

- `docs/phase-3-schema-changes.md` is directionally useful, but it is not an exact mirror of the current migration filenames or all current status/result semantics.
- `double_opt_in` and `gdpr_compliant` exist in the contract, but only GDPR is currently actionable from the editor and runtime. `double_opt_in` is not implemented end-to-end.
