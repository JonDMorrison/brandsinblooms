# v1.4.0 Notifications, Consent, And Automation Bridge

## Summary

The Form Builder notification story is broader than “email a team member.” The repo shows three connected layers:

1. consent and compliance capture on submission
2. form-specific staff notification queueing
3. automation trigger emission and worker processing after accepted submissions

This is also the area with the most obvious repo drift: the UI still contains a stale note implying notification routing is future work, but the worker code already queues notification messages.

## Consent And Compliance Capture

Primary files:

- `src/components/forms/FormComplianceTab.tsx`
- `src/lib/forms/fieldRegistry.ts`
- `supabase/functions/submit-form/index.ts`
- `docs/release-decision-form-builder-v1.md`

Confirmed behavior:

- email and SMS consent fields are first-class field types
- the editor lets operators configure the displayed consent text
- the submit pipeline stores canonical consent metadata on every submission
- customer consent is only upgraded, never downgraded
- `opt_out` is never modified by form submissions

Stored metadata includes:

- `email_consent`
- `email_consent_text`
- `email_consent_at`
- `email_consent_required`
- `sms_consent`
- `sms_consent_text`
- `sms_consent_at`
- `sms_consent_required`

## Notification Queueing On Accepted Submission

Primary worker file:

- `supabase/functions/process-form-submitted/index.ts`

Confirmed behavior in `queueFormSettingsNotifications(...)`:

- load the form by `tenant_id` and `form_id`
- read `settings_json.notification_emails`
- validate and normalize recipient emails
- fetch the related submission and customer details
- build a subject and body for the staff notification
- insert queued email work into `crm_outbox`

This means form-level staff notifications are not just stored. They are routed into the existing outbox infrastructure.

## Automation Trigger Bridge

Primary migrations:

- `supabase/migrations/20260128195905_f704b93c-a51d-4615-af1f-56c725658a44.sql`
- `supabase/migrations/20260129155218_5c689100-4821-43e8-bf26-a236a71a0b15.sql`
- `supabase/migrations/20260403120000_fb001_form_builder_contract_hardening.sql`

Primary worker file:

- `supabase/functions/process-form-submitted/index.ts`

Confirmed flow:

1. accepted `form_submissions` rows emit `form_submitted` events through `emit_form_submitted_event()`
2. events land in `automation_trigger_events`
3. `claim_trigger_events(...)` enables atomic worker claiming
4. `process-form-submitted` executes matching automations and also queues form-settings notifications
5. `increment_trigger_event_retry(...)` supports retry bookkeeping for failures

Supported automation action types called out in code:

- `send_email`
- `notify_staff`
- `delay`

## Downstream Delivery Dependency

The Form Builder code confirms queueing, not final inbox delivery.

The actual delivery path depends on the existing outbox processing infrastructure, including:

- `public.crm_outbox`
- `supabase/functions/process-automation-outbox/index.ts`

So the honest status is:

- notification queue creation is implemented in Form Builder flow
- final delivery relies on shared messaging infrastructure outside the Form Builder module itself
